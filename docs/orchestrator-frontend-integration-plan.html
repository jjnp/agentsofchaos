How the current graph frontend should plug into the orchestrator

The orchestrator already has the right high-level semantics for our current graph UI: global instance state, fork operations, integration-merge creation, artifacts, and a live event stream. The cleanest first step is not to replace the graph model, but to keep the existing canvas/domain layer and add a translation layer from orchestrator state + SSE events into that graph.

REST for initial reads + commands SSE as the source of truth for live updates Canvas placements remain frontend-owned Fork/merge mocks become orchestrator-backed actions
Best immediate fit

Our current merge visualization already matches orchestrator semantics surprisingly well: merge creates a new integration node rather than mutating the target in place.

Main translation gap

Backend identity is slot/session/agentUuid oriented. Frontend identity is UUID-only graph nodes. We should keep a frontend UUID and attach backend metadata, not overload slot as node id.

Critical real-time rule

Prompt completion must come from SSE pi_event.agent_end, not optimistic local state.

Recommended first principle

Preserve the graph domain. Add backend metadata beside it.

The graph is the product surface. The orchestrator is the execution backend. So the frontend should continue to render AgentNode and AgentNodePlacement, but enrich each node with a backend payload like slot / label / agentUuid / sessionId / artifact availability.

  • Good fit: current parent edge = fork lineage.
  • Good fit: current mergedNodes edge = source → integration merge edge.
  • Watch: current status enum is only running | completed; backend has richer lifecycle.
  • Watch: slots are global/process-local, so reconnect logic must re-bootstrap from GET /api/state.

Concrete mapping from orchestrator model to current graph model

Node identity AgentNode.id is UUID-only slot, sessionId, agentUuid Generate/stabilize a frontend UUID and store backend metadata separately so strict typing stays clean.
Parent relationship parentId fork_complete, fork responses, fork-point artifact When a fork produces target slot N from source slot M, create node N with parentId = node(M).
Merged nodes mergedNodes: AgentNodeId[] merge_integration_created / merge_complete, POST /api/merge Create a new integration node as a child of the target node; set mergedNodes = [sourceNodeId].

Context usage details.contextUsage fork-point artifact, future SSE-derived counters, merge artifacts Start by hydrating from fork-point artifact when present; for live nodes, allow unknown/zero until better telemetry exists.
Prompt composer Calls local fork() mock POST /api/instances/:slot/prompt Prompt should target the selected node’s backend slot and keep the same left-panel UX.

Merge action Mock integration node creation POST /api/merge Keep the exact UI metaphor; swap in backend merge call and wait for integration events.

Step-by-step integration plan

1

Introduce backend-aware graph types

Add a frontend-only extension beside AgentNode, e.g. AgentNodeRuntime or backend metadata.

  • slot
  • label
  • agentUuid
  • sessionId
  • artifact flags
2

Build a small orchestrator client layer

Create one thin module for REST and one for SSE, both typed from the documented payload shapes.

  • getState()
  • createInstance()
  • promptInstance()
  • forkInstance()
  • mergeInstances()
3

Add a reducer that turns backend state into graph state

This is the key seam: server payloads become our canvas model, without leaking transport concerns into Svelte components.

  • normalize initial state
  • upsert nodes by backend key
  • derive parentId from fork lineage
4

Replace prompt mock first

Smallest real backend win. Keep selection, left panel, and running spinner behavior; only the action changes.

  • send prompt to selected slot
  • stream output into node view
  • complete on SSE
5

Replace fork creation

Hook current “create child” UX to real fork semantics and artifact hydration.

  • use server-created target slot
  • fetch fork-point artifact
  • seed details context usage
6

Replace merge creation

The existing UI almost exactly matches backend semantics. This should be a natural swap after fork is working.

  • drag source onto target
  • call POST /api/merge
  • create integration child of target
  • show merged edge from source

What can stay exactly as-is

  • Canvas rendering, pan/zoom, layout modes, slide-over panels.
  • Parent edges and merged edges as visual primitives.
  • Node-focused left panel as the execution detail surface.
  • Placements and layout strategies staying fully client-side.
  • Selection as the main routing key for user actions.

What will likely need a small model evolution

  • Status model: eventually richer than running/completed.
  • Identity model: node UUID plus backend identity tuple.
  • Artifacts: per-node loading/error state for fork-point / merge-context / merge-details.
  • Reconnect correctness: SSE is buffered, but frontend should still re-bootstrap from /api/state.

Suggested first implementation slice

Slice 1: keep the existing canvas page, replace demo fixtures with GET /api/state + SSE bootstrapping, and wire only prompt execution for the selected node. That gives us a real backend-backed graph without yet needing to solve every lineage edge case. After that, fork and merge become incremental additions on top of the same reducer.
